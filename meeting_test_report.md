## 3. 测试剧本与流程 (The Round-based Script)

### 第 0 轮 (Round 0): 需求发起
- **输入:** Admin 发送邮件，请求讨论“新产品发布方案”，设定参与者为 GuestA 和 GuestB，截止时间设为明天。
- **初始提案:** “下周举办线上发布会，邀请媒体参加。”
- **预期结果:** Hub 识别为 `create_room` 动作，创建 `room_xxxx`，并给所有人发送带有协议附件的 CFP 邀请函。

### 第 1 轮 (Round 1): 意见分歧与冲突产生
- **成员反馈:**
  - **Admin:** 回复要求增加“用户抽奖环节”。
  - **GuestA:** 回复建议增加“北京办公室线下体验区”。
  - **GuestB:** 回复由于成本原因**反对**线下体验，但**赞成**抽奖。
- **Hub 行为:** Hub 收集全员邮件，分析出“线下体验”是一个冲突点，而“抽奖”是潜在共识。

### 第 2 轮 (Round 2): 妥协与收敛
- **成员反馈:**
  - **Admin:** 表示理解成本，同意放弃线下方案，锁定“线上+抽奖”。
  - **GuestA:** 表示同意放弃北京体验区。
  - **GuestB:** 再次确认接受。
- **Hub 行为:** 识别到全员 `ACCEPT` 状态。

### 第 3 轮 (Round 3): 决议出局 (Finalization)
- **预期结果:** Hub 自动关闭协商室，生成 Markdown 格式的正式纪要，并向所有人发送结案邮件。

## 4. 发现的问题

### A. 邮件服务商层面的频控封锁 (SMTP Error)
- **逻辑:** 当 Hub 试图在短时间内（同一秒）向多个参会者发出通知时，QQ 邮箱的 SMTP 服务器判定为高频骚扰，并强行切断连接 (`SMTPServerDisconnected`)。
- **影响:** 导致 Round 0 或 Round 1 的广播通知失败，测试剧本被迫中断。

### B. IMAP 搜索限制与效率低下
- **逻辑:** 国内邮箱（QQ/163）对 IMAP `SINCE` 指令支持不佳，Hub 启动时会尝试抓取数千封旧邮件进行重新解析。
- **影响:** 严重拖累 Hub 的响应速度，甚至造成解析旧指令导致的状态错乱。

### C. 设置边界模糊
- **逻辑:** 对于“帮我约个会”和“帮我开启内容协商”的语义解析有时会出现混淆。
- **影响:** 将原本应进入 Phase 2 的协商流程误判为了 Phase 1 的时间邀约。

## 5. 改进建议

1.  **引入发送队列与平滑延迟 (Smoothing):**
    在 `EmailClient` 发送环节增加一个缓冲队列，每封邮件之间间隔 2-3 秒，以规避反垃圾邮件系统的频控。
2.  **增强 UID 增量同步机制:**
    在代码中实现“只获取最后 N 封或从上次成功处理的 UID 开始”的逻辑，避免解析整个收件箱。已在 `lib/email_client.py` 中临时修复为 `uid_list[-50:]`。
3.  **强化 Room 模式的引导语建议:**
    在 `hub_prompts.py` 中对 `create_room` 增加明确的语义特征（如检测到关键词“讨论”、“方案”、“提案”、“截止时间”时自动触发）。
4.  **Loopback 测试优化:**
    优化 Hub 的身份识别逻辑，允许在特定配置下处理发给自己地址的邮件，方便单邮箱环境下的功能调测。
